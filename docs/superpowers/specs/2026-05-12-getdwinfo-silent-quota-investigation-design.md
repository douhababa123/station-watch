# GetDWinfo Silent Quota Investigation Design

**日期：** 2026-05-12  
**状态：** 已批准

---

## 背景

当前 GetDWinfo 在 `/homeappliances/{haId}` 与 `/homeappliances/{haId}/events` 上持续收到 `429`。结合 Home Connect 官方文档，这已经不像普通的分钟级限流，更像以下几类问题之一：

1. 同一 `client + user` 组合仍有隐藏请求源在持续消耗额度。
2. 同一 `client + user` 组合仍有隐藏 SSE 通道未释放。
3. 日额度或滚动额度窗口尚未恢复。

因此当前最需要的不是继续试接口，而是先把所有潜在的隐性流量来源系统性排空。

## 目标

1. 产出一份可执行的“彻底静默排查清单”。
2. 覆盖本机、浏览器、脚本、账号、OAuth Client、SSE 通道、token refresh、自动重试等全部隐性流量来源。
3. 明确静默窗口、验证顺序、对照实验和判定逻辑。
4. 让后续排查过程可重复、可交接，而不是继续依赖聊天记录。

## 非目标

1. 不修改当前 GetDWinfo 运行逻辑。
2. 不尝试在本次文档内直接解决 `429`。
3. 不新增自动化探针或新的排障接口。

## 方案

### 1. 定义“静默排查”边界

文档需要先明确：

1. 静默排查的目的，是确保 Home Connect 在一个完整观察窗口内不再收到任何非必要请求。
2. 观察窗口分为短窗口与长窗口：
   - 短窗口：覆盖 `1` 分钟限流与 `10` 分钟错误惩罚。
   - 长窗口：覆盖天级或滚动额度窗口。
3. 静默期结束前，不允许边看边试。

### 2. 按来源拆分排查清单

文档按以下来源逐项排查：

1. 本机进程与本地端口。
2. 浏览器标签页与自动重连页面。
3. VS Code 终端、调试任务、脚本与 Postman。
4. 账号与 OAuth Client 复用场景。
5. SSE 通道残留。
6. Token refresh 与自动恢复逻辑。
7. 自动重试与错误放大器。

### 3. 规定单点验证顺序

文档只允许以下验证顺序：

1. 先看本地认证状态。
2. 再打一条最小 `haId` 探活。
3. 仅在探活成功后才尝试单条 SSE。

避免一恢复就重新触发多条请求，导致排查失真。

### 4. 提供对照实验与判定逻辑

文档要给出三类对照：

1. 换账号。
2. 换 OAuth Client。
3. 换机器。

并把不同结果映射到更可能的根因类别，帮助判断是账号维度、client 维度还是本机环境问题。

## 预期结果

完成这份文档后，团队可以：

1. 用统一步骤确认是否还有隐藏流量源。
2. 在 `24h+` 后仍然 `429` 的情况下，快速切换到对照实验，而不是继续无效重试。
3. 为后续多账号、多设备、9 路 SSE 方案建立更可靠的上线前排障基线。
