# GetDWinfo Cache Export And Clear Design

**日期：** 2026-05-11  
**状态：** 已批准

---

## 背景与目标

`GetDWinfo/` 现在已经支持两层设备快照：

1. 浏览器 `localStorage` 快照
2. 服务端 `.appliances-cache.json` 磁盘缓存

为了更方便做 `429` 限流回归和问题复现，需要再补两类维护动作：

1. 导出当前快照为 `.json`
2. 分层清空浏览器快照或服务端缓存

本次目标是让用户可以明确知道自己操作的是哪一层缓存，而不是引入“自动猜测来源”的隐式行为。

---

## 方案选择

### 方案 A：一个统一按钮自动猜测来源

- 优点：按钮少，界面最简。
- 缺点：容易误导用户，不利于做限流回归时区分浏览器层与服务端层。

### 方案 B：一个按钮配下拉菜单

- 优点：界面仍比较紧凑。
- 缺点：多一步选择，且在频繁测试时不如独立按钮直观。

### 方案 C：每层缓存独立按钮

- 优点：来源清晰，适合排障和回归测试。
- 缺点：按钮数量更多。

### 最终选择

采用 **方案 C**。页面上直接提供四个独立动作：

1. `导出浏览器快照`
2. `导出服务端缓存`
3. `清空浏览器快照`
4. `清空服务端缓存`

---

## 后端设计

### 服务端缓存导出

新增接口：`GET /cache/appliances/export?source=server`

返回 JSON 结构，至少包含：

- `source`
- `cachedAt`
- `data.homeappliances`

如果服务端当前没有可用缓存，则返回 `404 cache_not_found`，而不是返回空对象。

### 服务端缓存清空

新增接口：`POST /cache/clear`

请求体示例：

```json
{ "scope": "server" }
```

服务端会：

1. 清空内存中的 `applianceListCache`
2. 删除磁盘 `.appliances-cache.json`
3. 返回最新诊断结果

浏览器层缓存不通过后端删除，避免引入对单个浏览器上下文不可见的伪语义。

---

## 前端设计

### 按钮布局

按钮直接放在现有“手工导入设备快照”卡片里，不新增第三张卡片。这样缓存工具仍集中在一块区域里，避免布局继续膨胀。

### 浏览器快照导出

前端直接从 `localStorage` 读取当前浏览器快照，生成 JSON Blob 并下载。下载文件名带上来源和时间戳，例如：

- `appliances-browser-snapshot-2026-05-11T12-34-56.json`

如果浏览器当前没有快照，则给出明确提示，不发起下载。

### 服务端缓存导出

前端调用 `GET /cache/appliances/export?source=server`，拿到 JSON 后下载。文件名同样带上来源和时间戳，例如：

- `appliances-server-cache-2026-05-11T12-34-56.json`

### 浏览器快照清空

前端只删除 `localStorage` 中的设备快照键，并立即刷新缓存诊断。

如果当前页面仍显示的是刚刚从浏览器快照渲染出的设备列表，不强行清空当前 UI，而是在状态条明确提示：

- 浏览器回退快照已删除
- 当前页若不刷新，仍会暂时保留已渲染列表
- 刷新后将不再有浏览器层回退能力

### 服务端缓存清空

前端调用 `POST /cache/clear` 且 `scope=server`。成功后刷新缓存诊断，并在状态条中说明服务端磁盘缓存已删除。

如果浏览器层快照仍在，当前页仍然可以继续依赖浏览器快照；如果浏览器层也为空，则下一次刷新将没有任何本地回退设备列表。

## 错误处理

### 服务端

- 没有缓存可导出：`404 cache_not_found`
- `scope` 非法：`400 invalid_cache_scope`
- 删除磁盘文件失败：`500 cache_clear_failed`

### 前端

- 浏览器没有快照：提示“当前浏览器没有可导出的设备快照”
- 服务端导出失败：显示接口返回的错误摘要
- 清空成功后：立即刷新缓存诊断并提示具体清掉的是哪一层
