# GetDWinfo Home Connect 彻底静默排查清单

**日期：** 2026-05-12  
**适用范围：** `GetDWinfo/` 本地代理、Home Connect 真实设备联调、账号/Client 配额排查  
**目的：** 在再次验证真实 `haId` 与 SSE 之前，系统性排空所有可能偷偷继续消耗 Home Connect 配额的来源

---

## 1. 什么时候需要执行这份清单

出现以下任一现象时，应停止继续试接口，转入静默排查：

1. `/homeappliances/{haId}` 单次探活仍返回 `429`。
2. `/homeappliances/{haId}/events` 单次 SSE 建链仍返回 `429`。
3. 已经等待 `24h` 左右，依然持续 `429`。
4. 不确定是否还有其它页面、脚本、机器、同事环境正在使用同一 `client + user` 组合。

---

## 2. 排查目标

本次排查的目标不是“尽快让页面跑起来”，而是先证明：

1. 在一个完整观察窗口内，Home Connect 不再收到任何非必要请求。
2. 当前 `client + user` 组合没有残留 SSE 通道。
3. 当前 `client + user` 组合没有残留 token refresh、自动重试、自动探活或自动恢复会话流量。

---

## 3. 观察窗口定义

### 3.1 短静默窗口

用于覆盖分钟级限流与连续错误惩罚：

1. 至少静默 `15` 分钟。
2. 期间不允许任何接口验证动作。

### 3.2 长静默窗口

用于覆盖日额度或滚动额度问题：

1. 至少静默 `24-30` 小时。
2. 期间避免任何真实 Home Connect 请求。

---

## 4. 本机侧排空清单

### 4.1 浏览器

1. 关闭所有打开过 `http://localhost:3000` 的标签页。
2. 关闭所有可能自动重连 SSE 的页面。
3. 最稳妥做法是整个浏览器退出，而不是只关标签。

### 4.2 进程与服务

1. 停掉 GetDWinfo 本地代理。
2. 停掉所有 PowerShell / Node / curl / Postman 测试脚本。
3. 停掉任何可能使用同一个 `client_id` 的内部小工具。
4. 确认本机没有残留 Home Connect 相关计划任务、自启程序或后台服务。

### 4.3 端口与终端

1. 确认 `3000` 端口没有残留监听。
2. 检查 VS Code 里所有终端，确保没有旧脚本仍在后台运行。
3. 检查是否有多个仓库目录或多个分支目录都启动过类似服务。

---

## 5. 环境与脚本排空清单

### 5.1 测试工具

1. 关闭 Postman collection runner。
2. 停掉任何 curl / PowerShell 循环调用脚本。
3. 检查是否有浏览器插件或内部工具在定时访问 Home Connect。

### 5.2 自动恢复与自动重试

重点确认以下逻辑没有在暗中继续触发请求：

1. 启动时自动 refresh token。
2. `401` 后自动 refresh 再重试。
3. 页面加载时自动拉设备列表。
4. SSE 失败后立即自动重连。
5. 认证状态为 `recoverable` 时自动恢复会话。

---

## 6. 账号与 OAuth Client 排空清单

Home Connect 的关键计数维度是：**同一个 `client + Home Connect user account`**。

因此必须逐项确认：

1. 同一账号是否在别的电脑上登录过。
2. 同一账号是否在同事机器上被联调。
3. 同一个 OAuth Client 是否被别的项目或脚本复用。
4. 是否存在同一账号、同一 Client、多个环境同时访问的情况。

如果这是 Bosch 内部项目，还要确认：

1. 是否有 CI、测试机、演示机在使用同一个 Client。
2. 是否有别的团队成员正在用同一个账号或同一个 Client 做实验。

---

## 7. SSE 通道排空清单

1. 关闭所有可能建立过 `/events/{haId}` 的页面与程序。
2. 不要只依赖本地 `activeSseChannels=0`；它只能代表当前这一个代理，没有代表其它机器或其它服务没有占通道。
3. 确认是否有异常断线后的自动重连逻辑仍在运行。
4. 对每个账号单独判断，不要把多个账号的 SSE 状态混在一起看。

---

## 8. Token 与错误放大器排空清单

### 8.1 Token refresh

官方 token refresh 也单独受限：

1. 每分钟最多 `10` 次。
2. 每天最多 `100` 次。

因此要确认没有以下隐藏循环：

1. access token 过期。
2. 自动 refresh token。
3. refresh 后立刻自动探活。
4. 探活失败又触发下一轮恢复。

### 8.2 错误放大器

官方文档说明：

1. `10` 分钟内连续 `10` 次错误请求，可能再封 `10` 分钟。

因此必须停掉所有“失败就再试”的逻辑，包括：

1. 自动重试。
2. 自动重连 SSE。
3. 页面打开就自动拉详情。
4. 失败后立即刷新页面。

---

## 9. 静默窗口期间的纪律

在观察窗口结束前，不要做以下动作：

1. 不要打开 `localhost:3000` 页面。
2. 不要点击登录。
3. 不要点刷新。
4. 不要探活 `haId`。
5. 不要手工建立 SSE。
6. 不要用 Postman / curl / PowerShell 验证真实接口。

---

## 10. 静默后单点验证顺序

静默窗口结束后，只允许按下面顺序验证，且每一步都要先看结果，再决定是否进入下一步：

### Step 1. 先看本地认证状态

1. 仅查看本地 `auth/status`。
2. 不要立刻打设备列表。

### Step 2. 只打一条最小探活

1. 仅验证一个已知真实 `haId`：`/debug/appliance-by-id?haId=...`
2. 不要同时试多个设备。

### Step 3. 仅在探活成功后尝试一条 SSE

1. 只对同一个 `haId` 建一条 SSE。
2. 不要立刻扩成多台多条通道。

---

## 11. 对照实验清单

如果完成长静默窗口后仍然 `429`，不要继续重复同一组验证，而应切换到对照实验。

### 11.1 换账号

1. 使用一个未被频繁联调的 Home Connect 账号。
2. 保持同一个 OAuth Client。
3. 仍然只打一条最小探活。

### 11.2 换 OAuth Client

1. 使用新的 OAuth Client。
2. 保持同一个账号。
3. 仍然只打一条最小探活。

### 11.3 换机器

1. 使用一台完全干净的机器。
2. 保持同账号、同 Client。
3. 目的在于排除本机残留脚本或环境噪音。

---

## 12. 结果判定逻辑

1. `15` 分钟静默后恢复：更像分钟级限流或连续错误惩罚。
2. `24-30h` 后仍不恢复，但换账号恢复：更像该账号维度仍有隐藏流量，或该账号的日额度窗口尚未恢复。
3. 换账号仍不恢复，但换 Client 恢复：更像当前 OAuth Client 被持续占用或已触发更重限制。
4. 换账号、换 Client、换机器都不恢复：需要怀疑更上层的账号配置、China 同步或官方侧限制，应考虑联系 Home Connect 官方支持。

---

## 13. 对多账号、多设备方案的特别提醒

如果后续规划是：

1. `6` 个账号。
2. 每个账号 `9` 台设备。
3. 每个账号 `9` 条 SSE。
4. 复用同一个 OAuth Client。

则联调阶段不要一上来就直接跑满。更稳妥的验证顺序是：

1. 先验证 `1` 个账号、`1` 台设备、`1` 条 SSE。
2. 再扩到 `1` 个账号、`3` 台设备。
3. 最后再扩到 `1` 个账号、`9` 台设备。
4. 单账号稳定后，再复制到多个账号。

---

## 14. 建议执行顺序

1. 先停掉所有 Home Connect 相关进程、页面、脚本。
2. 进入 `15` 分钟短静默窗口。
3. 若仍异常，再进入 `24-30h` 长静默窗口。
4. 长静默后先打一条最小探活，不要立刻试 SSE。
5. 如果仍然 `429`，立即切换为“换账号”或“换 Client”对照实验。

这份清单的核心原则只有一句：**先证明“没有隐藏流量”，再谈恢复验证。**
