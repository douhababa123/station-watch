# Home Connect 限流排障文档（中文）

**日期：** 2026-05-11

---

## 这份文档解决什么问题

当 GetDWinfo 页面出现 `429 Too Many Requests` 时，这并不一定表示“当前这一秒请求太多”。
根据 Home Connect 官方文档，`429` 可能来自以下几类限制：

1. 短时间请求速率过高。
2. 每分钟或每日总额度已用尽。
3. 并行 SSE 监听通道过多。
4. 最近 10 分钟内连续错误请求过多，触发临时封禁窗口。

这份文档把官方规则整理成中文，并对应到 GetDWinfo 里的可观测项，方便排查当前 `429` 的真实原因。

## 官方限流规则

以下内容整理自 Home Connect 官方文档与 OpenAPI 描述。

### 1. 常规 REST 请求限制

1. 平均 `10 requests / second`。
2. 峰值 `20 requests / second`。
3. `50 requests / minute`。
4. `1000 requests / day`。

这几条是最常见的配额限制。即使页面现在只发一条请求，如果你在此前几分钟已经反复刷新、重启服务、切换页签、重复登录或调试脚本，也可能已经把分钟级或日级额度打满。

### 2. SSE 事件监听通道限制

1. 同一个 client + Home Connect user account，最多 `10` 条并行 monitoring channel。
2. 典型报错会表现为事件接口返回 `429`，并提示并行监听过多。

这意味着如果多个浏览器页、多个本地服务实例、或者断开的旧 EventSource 没有及时释放，就可能把 SSE 通道额度占满。

### 3. Token 刷新限制

1. `10 token refresh / minute`。
2. `100 token refresh / day`。

如果本地工具进入异常恢复循环，重复刷新 token，也会单独消耗这一类额度。

### 4. 程序控制接口限制

1. `5 start-program / minute`。
2. `5 stop-program / minute`。

这类额度主要影响控制操作，不是当前只读页面的主因，但如果后续加控制功能，需要单独注意。

### 5. 连续错误请求惩罚规则

官方文档还有一条非常关键但容易忽略的规则：

1. 如果在 `10 分钟` 内，出现 `10` 次连续错误响应，请求可能会被额外阻断 `10 分钟`。

这意味着即使当前页面已经把请求数降到最低，只要之前的调试流量里出现大量 `401 / 404 / 429 / 5xx`，也可能进入临时封禁窗口。

## `Retry-After` 怎么看

当 `429` 来自时间窗口类限制时，服务端可能返回 `Retry-After` 响应头。

它的含义是：

1. 如果是秒数，表示建议等待多少秒后再试。
2. 如果缺失，不代表不是限流；有些按总量或内部风控触发的 `429` 不一定带这个头。

所以排障时应这样理解：

1. `Retry-After` 有值：说明上游明确给出了冷却时间。
2. `Retry-After` 为空：仍然可能是分钟额度、日额度、SSE 通道额度，或者连续错误惩罚窗口。

## 对 GetDWinfo 的具体影响

在当前 GetDWinfo 架构下，请求已尽量压缩为：

1. 首屏最多一条 `GET /homeappliances`。
2. 选中设备后优先建立一条 `/events/{haId}` SSE 通道。
3. `status / settings / raw / programs` 改为按需加载。

如果在这种前提下，冷启动第一条 `GET /homeappliances` 仍然是 `429`，通常更可能是以下原因之一：

1. 之前几分钟内累计请求过多，分钟额度还没恢复。
2. 当天累计请求已经接近日额度或超过日额度。
3. 之前调试过程中产生了较多连续错误请求，触发了 10 分钟惩罚窗口。
4. 仍有旧 SSE 通道或旧服务实例没释放，导致上游处于资源紧张状态。

## 推荐排障顺序

### 场景 A：首屏第一条 `/homeappliances` 就 429

优先检查：

1. 请求日志里最近一次 `GET /homeappliances` 的 `Retry-After`。
2. 最近 10 分钟错误次数是否已经接近或超过 10。
3. 当前 SSE 通道数是否不合理。
4. 是否刚刚做过频繁刷新、脚本联调或反复登录。

结论倾向：

1. 更像是分钟额度、日额度，或连续错误惩罚窗口。
2. 不太像“当前页面又多打了几条详情接口”，因为第一条冷启动请求已经失败。

### 场景 B：设备列表能出来，但选中设备或切 tab 后才 429

优先检查：

1. 是否在短时间内对同一设备反复切换 `status/settings/raw/programs`。
2. 是否多个标签页同时监听同一批设备事件。
3. 请求日志里是否出现程序相关接口集中爆发。

结论倾向：

1. 更像是页面交互触发的短时突发请求。
2. 应继续收敛页签切换时的补充请求节奏。

### 场景 C：SSE 事件接口本身 429

优先检查：

1. 当前活跃 SSE 通道数。
2. 是否存在旧浏览器页、旧本地服务、重复 EventSource。

结论倾向：

1. 更像是 `10` 条并行 monitoring channel 上限被占满。

## 官方建议与当前实现的关系

Home Connect 官方建议优先使用 monitoring functions，而不是高频轮询。当前 GetDWinfo 已经按这个方向收敛：

1. 设备选中后优先走 SSE。
2. 详情接口改为按需加载。
3. 浏览器快照和服务端缓存用于避免无意义重复请求。

但需要注意：

1. SSE 可以减少轮询，不代表 SSE 本身没有配额。
2. 缓存能减少请求，不代表已经进入的惩罚窗口会立即解除。

## 目前最实用的判断方法

排查 `429` 时，不要只看页面提示。要同时看这几项：

1. 请求日志里最近的真实上游接口。
2. `Retry-After` 是否存在。
3. 当前 SSE 通道数。
4. 最近 10 分钟错误次数。
5. 当前累计错误计数是否正在持续增长。

如果这几项都集中指向“冷启动第一条请求就失败”，那优先判断为上游额度窗口问题，而不是当前 UI 又写错了请求路径。

## 本仓库里的落地建议

1. 避免同时开多个 GetDWinfo 服务实例。
2. 避免多个浏览器标签页同时监听同一账号的事件流。
3. 不要把失败重试写成自动快速循环。
4. 保留请求日志和限流判因面板，优先根据真实上游记录判断问题。
5. 当 `Retry-After` 或最近 10 分钟错误次数明显异常时，先等待冷却窗口过去，再继续调试。
